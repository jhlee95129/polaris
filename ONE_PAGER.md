# 한수 — AI 사주 라이프 코치

## 1. 도메인 선택 이유

한국 사주/운세 시장 1.4조원. MZ세대가 전문 플랫폼 사용자의 80%. AI 사주 이용률(40.7%)이 오프라인 점술(18.5%)을 이미 압도. 그러나 기존 서비스(점신, 포스텔러, 플롯)는 모두 **"해석 리포트 자판기"**에 머물러 있다. 사주라는 자기 이해의 프레임워크를 AI 라이프 코치의 렌즈로 재정의한다.

## 2. 타겟과 문제

**타겟**: 2030 직장인. 사주앱을 2-3개 병행하며 "다 비슷하다"고 느끼는 사람. 사주를 보고도 "그래서 나는 뭘 해야 하지?"라는 질문이 남는 사람.

**문제 3가지**:
1. **일회성 소비** — 프로필 보고 상담 한두 번 하고 끝. 매일 돌아올 이유가 없다.
2. **해석-행동 단절** — "올해 관재수 조심" 같은 해석은 있지만, 구체적으로 뭘 하라는 건지 알 수 없다. 사용자가 원하는 건 ①감정 확인 ②행동 허가 ③구체적 방향인데, 기존 앱은 ①만 겨우 해준다.
3. **빈 textarea 공포** — ChatGPT 사주는 매번 생년월일 재입력, "뭘 물어야 하지?"에서 이탈. 자유 입력은 진입장벽이 높다.

## 3. 핵심 기능 — 세 가지 킬 기능

**F1. "오늘의 한수"** — 매일 바뀌는 맞춤 행동 카드

앱을 열면 3초 내에 오늘의 일진 + 내 원국 기반의 **구체적 행동 1가지**를 제시. "올해 금전운이 좋아요"가 아닌 "오늘 점심에 평소 말 안 하던 동료에게 먼저 커피를 제안해보세요". 매일 바뀌는 콘텐츠가 재방문 루프를 만든다. → **일회성 소비 문제 해결**

**F2. 선택형 상담 플로우** — 채팅이 아닌 구조화된 코칭 경험

빈 textarea 대신 `카테고리(4) → 세부 상황(4) → 긴급도(4) → 자유입력(선택)` 4단계 선택형 플로우. 탭 몇 번만으로 상담 시작. 구조화된 맥락 → Claude에 더 좋은 질문 → 더 정확한 코칭 카드(진단/한수/타이밍/피할 것/근거). → **빈 textarea 공포 해결 + 해석-행동 단절 해결**

**F3. 반복 코칭 사이클** — "해봤어요" 후속 코칭

기존 사주 앱은 1회 상담으로 끝나지만, 한수는 **코칭 → 실행 → 보고 → 후속 코칭** 루프를 제공. 사용자가 한수(행동)를 실행한 뒤 "해봤어요"로 결과를 보고하면, 이전 맥락을 이어서 다음 단계의 행동을 코칭. 같은 고민에 대해 점진적으로 깊어지는 상담 경험. → **일회성 소비 → 지속적 코칭 관계로 전환**

**F4. 기운 시스템** — 자연스러운 모네타이제이션

상담 1회 = 기운 1개 소모. 매일 3개 무료 충전. 기운이 떨어지면 충전(결제) 유도. 오늘의 한수는 무료. "무료 콘텐츠로 습관화 → 유료 상담으로 전환"하는 프리미엄 모델.

## 4. 성공 지표

| 지표 | 목표 | 측정 방법 | 실패 시 액션 |
|---|---|---|---|
| **일일 재방문율** (북극성) | ≥ 40% | "오늘의 한수" 일일 호출 수 / DAU | 한수 카드 품질 이터레이션, 푸시 알림 |
| 세션당 상담 수 | ≥ 1.5 | 상담 완료 수 / 세션 | 선택형 플로우 최적화, 카테고리 추가 |
| 후속 코칭 전환율 | ≥ 25% | "해봤어요" 클릭 수 / 결과 표시 수 | 후속 코칭 CTA 개선, 알림 추가 |
| 기운 소진 후 충전율 | ≥ 15% | 충전 클릭 수 / 기운 0 도달 수 | 무료 기운 수 조정, 가격 테스트 |
| 피드백 응답률 | ≥ 20% | 피드백 제출 수 / 상담 완료 수 | 피드백 타이밍/UX 변경 |

## 5. 기술 하이라이트

- **만세력 라이브러리** (KASI 기반): LLM 확률이 아닌 정확한 사주팔자 계산
- **Claude tool_use 3종**: 코칭 카드 / 프로필 요약 / 오늘의 한수 — 구조화 출력으로 UX 품질 보장
- **4계층 시스템 프롬프트**: 명리학 지식 → 오늘 천기 → 사용자 원국 → 코칭 스타일
- **후속 코칭 프롬프트**: 이전 코칭 카드 + 사용자 실행 보고를 컨텍스트로 주입하여 맥락 연속성 보장
- **RAG**: 명리학 고전 → pgvector → 근거 기반 코칭 (프롬프트 하드코딩 대신 확장 가능)
- **선택형 플로우 빌더**: 구조화 맥락 → 자연어 질문 변환 → Claude에 전달

## 6. 의사결정 기록

### 왜 캐릭터를 제거했나?
초기에 캐릭터 3종(친한 언니/형, 할머니 역술가, 팩트체커)을 설계했으나, 사용자가 "어떤 캐릭터가 더 나은 조언을 해주는지" 판단할 수 없는 불필요한 선택지라고 판단. 캐릭터 선택은 마찰만 증가시키고 서비스의 핵심 가치(행동 코칭)에 기여하지 않음. 단일 "따뜻한 코치" 보이스로 통합하여 일관된 브랜드 경험 제공.

### 왜 반복 코칭인가?
사주 앱의 근본적 한계는 "1회 소비"다. 해석을 읽고 끝. 반복 코칭 사이클은 사용자를 "읽는 사람"에서 "실행하는 사람"으로 전환시킨다. 실행 → 보고 → 후속 코칭의 루프가 자연스러운 리텐션 메커니즘이 된다.
